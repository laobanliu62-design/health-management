# 📋 来福项目管理 SOP

> **目标**: 从4.5/10包工头 → 8/10项目经理

---

## 1. 任务状态看板

每个任务严格三态流转：

```
📋 todo → 🔧 doing → ✅ done
                  → 🚨 stuck
```

### stuck 升级机制
- **stuck 30分钟** → 来福主动介入排查
- **stuck 1小时** → 汇报老板，建议换人/换方案
- **stuck 2小时** → 强制重新分配或降级任务

### 看板文件
`hiclaw-fs/shared/projects/{project}/kanban.md`

格式：
```markdown
| 任务 | Worker | 状态 | 开始时间 | 卡住时间 | 备注 |
|------|--------|------|----------|----------|------|
| T1 | flutter-dev | ✅ done | 04-09 10:50 | - | 6min完成 |
| T9 | test-runner | 🚨 stuck | 04-12 00:20 | 2h+ | 22编译错误 |
```

## 2. 每日站会（心跳驱动）

每次心跳执行：

### 三句话站会
1. **昨天完成了什么** — 汇总done的任务
2. **今天要做什么** — todo→doing的任务
3. **有什么卡点** — stuck的任务+升级状态

### 输出格式
```
📢 站会 2026-04-17 10:30 北京时间
✅ 完成: T1-T7 (flutter-dev)
🔧 进行: T8 (flutter-dev, 开始30min)
🚨 卡点: T9 (test-runner, stuck 2h, 已介入)
📊 负载: flutter-dev 80% / qa-tester 0% / doc-writer 0%
```

## 3. 负载均衡

### 原则
- 单Worker不超过3个并行任务
- 闲置Worker超过2天 → 主动申请任务或回收
- 紧急任务优先分配给负载最低的Worker

### 负载监控
```
📊 Worker负载
flutter-dev:     ████░░░░ 50% (2/4 tasks)
doc-writer:      ░░░░░░░░  0% (闲置2天⚠️)
qa-tester:       ██░░░░░░ 25% (1/4 tasks)
docker-expert:   ░░░░░░░░  0% (闲置3天🔴)
```

## 4. 问题升级 SOP

```
Level 0: Worker自己解决 (0-15min)
Level 1: 来福介入排查 (15-30min)  
Level 2: 来福修复/重分配 (30min-1h)
Level 3: 汇报老板决策 (1h+)
```

### Worker卡住的标准处理
1. 读取Worker日志，定位问题
2. 能修的直接修（拼写错误、类型错误等）
3. 修不了的重分配给其他Worker
4. 都搞不定 → Level 3 汇报

## 6. 项目启动流程

新项目标准步骤：
1. **需求文档** — requirements.md (一句话+核心流程+成功标准)
2. **任务拆解** — tasks.md (每个任务可独立交付)
3. **看板初始化** — kanban.md (全部todo)
4. **Worker分配** — 负载最低优先
5. **启动通知** — 告诉老板开干了
